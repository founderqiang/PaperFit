# /adjust-length — 控制页数，但先修版面再碰语义

## 这是什么

`/adjust-length` 用于把论文调整到目标页数或目标页数变化。用户表达的是 “压到 8 页” 或 “补到 10 页”，PaperFit 负责自动判断先用排版手段还是最后才动语义，并把视觉结果作为是否达标的核心依据。

## 用户怎么说

以下表达都应进入长度调整任务：

- `/adjust-length =9`
- `用 PaperFit 把正文压到 8 页`
- `尽量不改内容，把这篇论文缩短 1 页`
- `增加半页到一页，让末页别这么空`

## Agent 自动完成什么

1. 自动识别当前页数、目标页数和页差。
2. 自动优先使用排版级手段：
   - 空间利用优化
   - 浮动体重排
   - 表格重构
   - 局部溢出修复
3. 只有在这些排版手段用尽后，才允许进入受控语义修改。
4. 每一轮长度调整后都重新编译并回到页图验收，确认不是“页数达标但视觉变差”。

## 用户会得到什么

- 当前页差是否已消除
- 为达到目标使用了哪些排版策略
- 是否已经触发语义修改，触发原因是什么
- 达标后是否仍存在残余视觉问题或语义风险

## 视觉与语义边界

- 页数达标不等于任务完成，仍必须视觉复验
- 默认先用排版手段，后用语义手段
- 语义修改必须最小、受控、可审计
- 包含 `Figure/Table/\ref{}` 锚点的导航句不得作为压字数候选

## 默认推断

- 用户只说“压一页”时，自动从当前 PDF 推断目标值
- 用户未说明是否允许语义修改时，默认仅在排版手段用尽后才谨慎触发
- 用户未指定优先章节时，优先选择风险较低的非核心叙述区

## 调度映射

- 任务类型：`adjust_length`
- 主调度：`orchestrator-agent`
- 触发保护：语义修改只能在明确满足触发条件后调用 `semantic-polish-agent`
