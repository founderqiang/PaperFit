# /check-visual — 只做视觉诊断，不改源码

## 这是什么

`/check-visual` 是一个视觉优先的诊断入口。用户只需要表达“先看问题，不要改代码”，PaperFit 就会自动完成编译、页图渲染、逐页排版检查，并返回结构化视觉结论。

它适合这些场景：

- 先判断当前论文有哪些排版问题
- 在自动修复前做一次基线诊断
- 人工改过源码后，只想重新验收视觉结果

这条入口的目标不是让用户手动跑检测脚本，而是让 Agent 代替用户完成一次完整的视觉检查闭环。

## 用户怎么说

以下表达都应路由到同一类任务：

- `/check-visual`
- `用 PaperFit 检查这篇论文现在的排版问题`
- `先做视觉诊断，不要自动改 tex`
- `帮我看一下哪些页还有表格、浮动体或溢出问题`

## Agent 自动完成什么

1. 自动识别主 `.tex` 文件、模板类型和单双栏形态。
2. 必要时自动重新编译，确保诊断基于当前 PDF。
3. 自动渲染页图，并逐页执行视觉检查。
4. 结合日志、crossrefs 和页图证据，识别空间利用、浮动体、一致性、溢出与模板迁移相关问题。
5. 输出视觉诊断摘要与结构化报告，但**不写回源码**。

## 用户会得到什么

- 一份以视觉问题为中心的诊断结论
- 每类问题所在页码、对象、严重程度
- 是否建议进入完整修复闭环
- 若暂不修复，当前主要风险是什么

典型输出重点应包括：

- 哪些页存在明显视觉缺陷
- 哪些 figure/table 远离对应语义段落
- 是否存在页数超标、末页留白、公式或表格溢出
- 是否发现影响交付的 Major/Critical 问题

## 边界

- 这条入口不直接改 `.tex`
- 不因为“只是检测”而跳过页图渲染
- 没有视觉检查证据时，不得报告“排版正常”
- 日志和源码只作为辅助证据，不能替代视觉结论

## 默认推断

- 未显式指定主文件时，优先自动识别包含 `\documentclass` 的主文件
- 未显式指定页码范围时，默认检查全文
- 未显式指定输出位置时，报告写入标准运行时产物路径

只有在主文件无法可靠判断、编译环境损坏或权限不足时，Agent 才应向用户补充提问。

## 调度映射

- 任务类型：`visual_only`
- 主责 Agent：`layout-detective-agent`
- 最终输出：视觉诊断报告 + 面向用户的缺陷摘要
