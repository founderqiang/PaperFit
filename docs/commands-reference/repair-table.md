# /repair-table — 修单个表格，但仍走视觉闭环

## 这是什么

`/repair-table` 用于处理单个表格或一小组明确表格的问题。用户可以只指出 “Table 3 很挤” 或 “修一下 `tab:ablation`”，PaperFit 会自动定位对应源码、分析表格缺陷、执行局部修复，并回到页图上确认结果。

它不是一个“只改几行 TeX 就结束”的命令，而是一个局部对象修复入口。

## 用户怎么说

以下表达都应触发同一任务：

- `/repair-table tab:ablation`
- `用 PaperFit 修一下 Table 3`
- `这个表格太挤了，帮我修到能正常投稿`
- `只处理 Table 1，不要动整篇其他部分`

## Agent 自动完成什么

1. 自动识别用户指的是表格编号、`\label{}` 还是页图里的目标对象。
2. 自动在 `.tex` 中定位对应 `table` / `table*` 环境与相关上下文。
3. 自动检查表格是否存在这些问题：
   - 列宽失衡
   - 对齐混乱
   - 行间距和横线距离过紧
   - 视觉风格与其他表不一致
   - 溢出、过窄、过宽或错误浮动位置
4. 自动选择合适的局部修复方式，例如宽度感知重构、列格式归一化、局部间距调整、位置参数修正。
5. 自动重新编译并回到页图确认该表是否真正变好，而不是只看源码 diff。

## 用户会得到什么

- 该表格的问题摘要
- 实际采用的修复策略
- 修复后的视觉结果是否通过
- 是否引入了残余风险，例如仍偏晚出现、仍与正文距离过远、仍有轻微不均匀

## 边界

- 不允许通过 `\resizebox`、`\scalebox` 暴力压缩表格
- 不允许通过改变模板字体包或整体字体观感来“修复”表格
- 不允许删除 caption、label、表格主体或引用锚点句
- 单表修复完成后，仍必须回到视觉闭环复验

## 默认推断

- 用户只说 `Table 3` 时，优先通过 crossrefs 和 caption 解析做自动映射
- 用户只给出“这一页的表格”时，允许结合当前页图与编号做对象识别
- 未提供保守性要求时，默认优先保持表格语义、模板字体和整体风格不变

只有在同名对象歧义明显时，才向用户发起最小化澄清。

## 调度映射

- 任务类型：`repair_table`
- 主责执行：`code-surgeon-agent`
- 验收要求：必须包含修复后视觉复验结果
